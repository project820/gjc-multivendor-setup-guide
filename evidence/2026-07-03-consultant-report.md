## 📋 GJC Ultimate Setup — Development Consulting Report

### Idea Summary
The GJC "Ultimate" line is a four-provider, role-based coding-agent profile (`ultimate`, `ultimate-f5`, `legend`) that routes `default`/`executor` to Anthropic Claude, `planner` to OpenAI GPT-5.5, `architect` to Google Gemini 3.1 Pro, and `critic` to xAI Grok 4.3, all defined in [`gjc-profiles.yml`](https://github.com/project820/gjc-multivendor-setup-guide/blob/main/gjc-profiles.yml) on top of the [`gajae-code` (GJC)](https://github.com/Yeachan-Heo/gajae-code) harness.

> **Editorial note (2026-07-03, added at publication):** this report treats a post-launch "Opus 4.8 tool-loop regression" (citing [Ready Solutions](https://readysolutions.ai/blog/2026-05-31-opus-4-8-tool-state-regression/)) as an established risk in §2, §4 and the recommendations. The companion [deep-research-benchmarks.md](./2026-07-03-deep-research-benchmarks.md) rates this **low-confidence / community-only** — it found *no official Anthropic acknowledgement* of an Opus-4.8-specific regression. Treat the external-CI-verification recommendations as prudent defense-in-depth, **not** as confirmation of a vendor-acknowledged defect. (Per MAINTAINING.md, published evidence is not rewritten; this note reconciles the two reports at publication.)

### 1. Landscape Analysis
- **Similar tools**: Cursor (whose Composer 2.5 model family is itself debated as a GJC Executor candidate — [xAI Composer 2.5 announcement](https://x.ai/news/composer-2-5)), Continue, Aider, Cline, and OpenAI Codex CLI all occupy the same "agentic coding harness" category, but none of them ship a native four-role (planner/executor/critic/architect), four-vendor synchronous profile out of the box the way GJC does via its [role prompts](https://github.com/Yeachan-Heo/gajae-code/blob/main/packages/coding-agent/src/prompts/agents/executor.md).
- **Market trend**: The industry is converging on two ideas simultaneously — (1) **cross-model verification** (using an independent model to critique/judge another model's output to reduce self-preference bias, per [Self-Preference Bias in LLM-as-a-Judge](https://arxiv.org/abs/2410.21819) and [Judging with Many Minds](https://arxiv.org/abs/2505.19477)), and (2) **adaptive test-time compute** rather than uniform maximum reasoning effort, shown to yield over 4x efficiency gains in [Snell et al.'s scaling paper](https://arxiv.org/abs/2408.03314). GJC's multivendor pack is an early, community-built attempt at (1) but has not yet adopted (2).
- **Differentiation opportunity**: Nearly every competing harness either locks users into a single vendor (Codex CLI → OpenAI, Cline defaults → Anthropic) or treats multi-model use as manual copy-paste. GJC's role-scoped, YAML-declared multivendor profile is a genuine differentiator — the opportunity is to pair that differentiation with the adaptive-effort and failover discipline the current setup still lacks, per the [three-model council synthesis](https://github.com/project820/gjc-multivendor-setup-guide) analysis of this exact repo.

### 2. Configuration Feasibility Assessment

| Evaluation Item | Grade | Description |
|-----------------|-------|--------------|
| Implementation difficulty | ⭐⭐ | The YAML profile mechanism is already built and installed; changes are config-only edits to `~/.gjc/agent/models.yml`, not new engineering — low difficulty to iterate. |
| Technical risk | ⭐⭐⭐⭐ | Multiple open risks: the `gemini-3.1-pro-low:high` selector may be a silent bug workaround that under-powers the Architect ([selector evidence](https://github.com/project820/gjc-multivendor-setup-guide/blob/main/evidence/2026-07-02-selectors.md)); GJC issues document routing drift, silent model substitution, and stream-timeout misclassification ([issue #947](https://github.com/Yeachan-Heo/gajae-code/issues/947), [issue #543](https://github.com/Yeachan-Heo/gajae-code/issues/543), [issue #1368](https://github.com/Yeachan-Heo/gajae-code/issues/1368)); and Opus 4.8 has a documented post-launch tool-loop regression that undermines trust in Executor self-verification ([Ready Solutions report](https://readysolutions.ai/blog/2026-05-31-opus-4-8-tool-state-regression/)). |
| Scalability | ⭐⭐⭐ | The four-provider, role-scoped design scales conceptually to more roles or profiles, but four synchronous providers in one cycle multiply outage and rate-limit exposure, especially since Anthropic and OpenAI show correlated peak-hour latency spikes (Gemini 3.1 Pro council finding, corroborated by [Downdetector-style patterns](https://www.frankx.ai/blog/grok-4-3-analysis-2026)). |
| Maintainability | ⭐⭐⭐ | Config is centralized and versioned in one YAML file, which is good hygiene, but four vendor auth flows, model-catalog drift ([issue #1332](https://github.com/Yeachan-Heo/gajae-code/issues/1332), [issue #965](https://github.com/Yeachan-Heo/gajae-code/issues/965)), and an unresolved `grok-build` provider design ([Grok Build provider design](https://github.com/Yeachan-Heo/gajae-code/blob/main/docs/grok-build-provider-design.md)) add ongoing operational overhead. |

### 3. Recommended Role→Model Assignments

**default**: Keep `anthropic/claude-opus-4-8:high` (or `claude-fable-5:high` in the Fable line). This role sets the main-loop tone and doesn't need frontier max effort by default; `:high` is sufficient and all three council models implicitly converge on Claude leading the hardest coding benchmarks ([Vals AI SWE-bench](https://vals.ai/benchmarks/swebench)).

**executor**: Keep `anthropic/claude-opus-4-8` as the core, but change the default effort from `:max` to `:high`, escalating to `:max` only when the Critic returns ITERATE/REJECT. Reject the community's proposed swap to `xai/grok-composer-2.5-fast` — it lacks `supportsReasoningEffort` (breaking GJC's self-verification contract), caps at 200K context vs. Claude's 1M, and per [Cursor's own tier documentation](https://lushbinary.com/blog/composer-2-5-cost-optimization-fast-vs-standard-tier/), "Fast" is the identical model to "Standard" at roughly 6x the price. If throughput is genuinely needed for low-risk, reversible changes, add a separate `ultimate-fast` profile using Composer 2.5 **Standard**, never Fast.

**planner**: Keep `openai-codex/gpt-5.5`, but drop the always-on `:xhigh` to `:high` as the default, reserving `:xhigh` for cases with planning ambiguity or after a failed Critic pass. GPT-5.5 leads Terminal-Bench 2.0 (~83–85%), which matches the Planner's long-horizon sequencing needs ([Terminal-Bench 2.0 leaderboard](https://www.tbench.ai/leaderboard/terminal-bench/2.0)), but running maximum effort on every cycle is an over-spend the [test-time-compute scaling research](https://arxiv.org/abs/2408.03314) argues against.

**architect**: Verify the current `google-antigravity/gemini-3.1-pro-low:high` selector against a direct `gemini-3.1-pro:high` call — the `low` prefix may be an unintended selector-parsing workaround that silently under-powers this seat ([selector evidence](https://github.com/project820/gjc-multivendor-setup-guide/blob/main/evidence/2026-07-02-selectors.md)). This is the most capability-constrained seat in the entire lineup (SWE-bench Pro ~54%), yet Gemini's 1M–2M context remains uniquely valuable for whole-repository architecture reviews — so the fix is verification and selective escalation to Opus/GPT-5.5 for small, high-risk diffs, not a wholesale replacement.

**critic**: Keep `xai/grok-4.3:high` as the default — its cross-family independence from both the Anthropic Executor and OpenAI Planner is the strongest single architectural argument for its seat, and it ranks top-2 on HalBench sycophancy pushback ([HalBench v2](https://www.reddit.com/r/LocalLLM/comments/1u6y82j/halbench_29_oss_models_tested_on_a_custom_built/)), despite trailing on raw Artificial Analysis Intelligence Index (53 vs. GPT-5.5's 60 and Opus's 61 — [Artificial Analysis leaderboard](https://artificialanalysis.ai/leaderboards/models)). Add a `gpt-5.5:high` second-pass critic specifically for merge-critical or irreversible changes, following the meta-judge aggregation pattern shown to be more robust than single-critic or debate-style designs ([Judging with Many Minds](https://arxiv.org/abs/2505.19477)).

### 4. Migration Roadmap

**Phase 1 — Verify & Escalate** (est. 3–5 days)
- [ ] Change `executor` default effort from `:max`→`:high` and `planner` from `:xhigh`→`:high` across `ultimate`/`ultimate-f5`/`legend`, with an explicit escalation rule triggered by Critic ITERATE/REJECT or Architect BLOCK verdicts ([critic prompt](https://github.com/Yeachan-Heo/gajae-code/blob/main/packages/coding-agent/src/prompts/agents/critic.md), [architect prompt](https://github.com/Yeachan-Heo/gajae-code/blob/main/packages/coding-agent/src/prompts/agents/architect.md))
- [ ] Run a controlled A/B test of `google-antigravity/gemini-3.1-pro-low:high` vs. `gemini-3.1-pro:high` on a fixed architecture-review task set to confirm or rule out the selector-downgrade hypothesis
- [ ] Pin an external, out-of-model verification step (test runner or CI gate) after every Executor pass, rather than trusting Opus 4.8's self-reported verification, given its documented tool-loop regression ([Ready Solutions report](https://readysolutions.ai/blog/2026-05-31-opus-4-8-tool-state-regression/))

**Phase 2 — Second-Pass Critic & Failover** (est. 5–7 days)
- [ ] Add a `gpt-5.5:high` second-pass critic step, invoked only for merge-critical or irreversible changes, with the parent process aggregating both critics' verdicts rather than chaining them into a debate
- [ ] Define an intra-seat failover model for each role (a same-role fallback from a different provider that triggers on HTTP 429 or timeout), preserving both cross-vendor independence and uptime resilience

**Phase 3 — Hardening & Observability** (est. 7–10 days)
- [ ] Stand up a per-cycle cost/latency dashboard broken out by role and effort tier, using the estimated per-cycle costs (~$2.96 for `ultimate`, ~$5.06 for `ultimate-f5`, ~$3.56 for `legend`) as a baseline to measure the savings from the effort ladder
- [ ] Ship the optional `ultimate-fast` throughput profile on Composer 2.5 **Standard** (never Fast) for reversible, low-risk changes, and monitor GJC's still-unresolved `grok-build` provider design for stability before relying on it in production ([Grok Build provider design](https://github.com/Yeachan-Heo/gajae-code/blob/main/docs/grok-build-provider-design.md))

### 5. Overall Verdict
The Ultimate line is a competent, largely merit-driven multivendor architecture, not diversity theater — routing the Executor to Anthropic (SWE-bench Pro leadership), the Planner to OpenAI (Terminal-Bench 2.0 leadership), and the Critic to xAI (cross-family independence) each pass a reasonable justification test. Its biggest strength is architectural: a role-scoped, config-driven multi-model setup that most competing coding harnesses don't offer natively.

Its weaknesses are concrete and fixable, not fundamental. The setup burns cost by defaulting every seat to maximum reasoning effort (`:max`/`:xhigh`) on every cycle regardless of task difficulty — an empirically suboptimal pattern the test-time-compute literature explicitly warns against ([Scaling LLM Test-Time Compute](https://arxiv.org/abs/2408.03314)). The Architect seat is quietly the weakest link, both in raw capability (~54% SWE-bench Pro) and in a possible selector bug that has never been verified. And the four-provider synchronous chain, while defensible for independence, has more failure surface than necessary without any failover policy — a real operational risk given documented GJC issues around routing drift and silent model substitution ([issue #947](https://github.com/Yeachan-Heo/gajae-code/issues/947), [issue #543](https://github.com/Yeachan-Heo/gajae-code/issues/543)).

The two community debates driving scrutiny of this setup — "should Grok 4.3 be the Critic" and "should Composer 2.5 Fast replace the Executor" — are both lower-leverage than they appear. Keep Grok as Critic; reject Composer Fast as Executor. The real leverage is in cost discipline, Architect verification, and failover, none of which require swapping a single model.

**Success probability**: High (roughly 80%) that the recommended changes ship cleanly, since they are config-level edits to an already-working profile system rather than new engineering — the main execution risk is operational discipline in rolling out the escalation ladder and failover logic rather than technical feasibility.

### 6. Next Steps
1. Edit `gjc-profiles.yml` to change `executor` and `planner` default efforts from `:max`/`:xhigh` to `:high`, with escalation triggers wired to Critic/Architect verdicts.
2. Run a side-by-side test of `gemini-3.1-pro-low:high` against `gemini-3.1-pro:high` to resolve the Architect selector question before making any Architect-seat decision.
3. Add a `gpt-5.5:high` second-pass critic for merge-critical changes and document the aggregation rule (parent aggregates both critic verdicts, no debate chaining).
4. Pin an external CI/test-runner verification gate after every Executor run to compensate for Opus 4.8's known self-verification regression.
5. Track per-cycle cost before and after the effort-ladder change to confirm the expected savings relative to the ~$2.96–$5.06 baseline per cycle.
